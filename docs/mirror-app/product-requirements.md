# Mirror App 产品需求文档（PRD）

> 版本：1.0-draft
> 日期：2026-07-21
> 产品阶段：工程化 MVP → 受控家庭试点
> 默认市场假设：新加坡、provider-managed home monitoring

## 1. 产品定义

Reflexion Mirror 是固定在家庭镜面设备上的语音与视觉交互 Agent。它有两个明确分离的工作模式：

1. **日常陪伴与事务协助**：通过唤醒词进入自然对话，回答实时信息问题，执行受控工具调用，提供日程和用药提醒，并把需要 caregiver 介入的事项形成任务。
2. **每日认知状态采集**：通过短时、自然但协议化的对话采集语言、定向、回忆、叙事、日常功能、交互节奏和质量信号；后端结合个人历史生成纵向变化提示。

产品不向患者提供独立的痴呆诊断，不从外貌推断认知状态，不自行更改药物、剂量或医嘱。面向临床或研究的风险输出必须经过质量门控、版本追踪和专业人员复核。

## 2. 产品目标

### 2.1 MVP 目标

- 老年用户无需操作手机即可通过目标唤醒词开始对话。
- 用户可查询真实天气、日期、时间和已授权日程。
- caregiver 可配置用药计划；镜面端负责提醒、确认、记录漏服或拒绝，不负责处方决策。
- 每日 check-in 可稳定完成、断点保存并可靠上传。
- caregiver 可看到配对状态、设备状态、完成情况、简化摘要和需要跟进的事项。
- 系统能够积累质量合格的个人基线，并在证据充分时生成纵向变化提醒。
- 每条评估结果可追溯到会话、同意版本、输入质量、模型、prompt、规则和软件版本。

### 2.2 非目标

- 面向消费者的独立痴呆、MCI 或阿尔茨海默病诊断。
- 自动调整药名、剂量、频率或处理药物相互作用。
- 在没有 caregiver/provider 复核的情况下自动报警医疗机构。
- 以摄像头外观、年龄或表情直接判定痴呆。
- 在没有足够个人基线和质量覆盖时输出确定性长期趋势。

## 3. 用户与权限

| 角色 | 核心任务 | 可见信息 | 不应获得的能力 |
|---|---|---|---|
| Patient | 唤醒、对话、确认提醒、完成 check-in | 当前任务、简单反馈、隐私提示 | 原始风险分数、诊断标签、他人资料 |
| Caregiver | 配对设备、配置计划、查看依从性和待办 | 简化摘要、提醒记录、设备与会话状态 | 未经批准的临床结论、其他 caregiver 的患者 |
| Provider/Reviewer | 审核评估、趋势、质量和升级 | 完整证据、版本、QC、复核记录 | 无审计地修改历史结果 |
| Support/Admin | 设备诊断、版本和连接支持 | 最小必要设备遥测 | 默认访问会话正文和健康信息 |
| System Service | 执行工具、分析、通知 | 按服务职责最小授权 | 跨租户访问或无审计写入 |

## 4. 核心模式

### 4.1 Companion 日常助手

进入方式：目标唤醒词、镜面触控、caregiver 远程触发的非紧急任务。

能力：

- 普通话和英语为首发语言；其他语言必须分别完成 ASR、TTS、对话和评估验证后开放。
- 实时天气：读取设备/患者授权位置或 caregiver 配置城市，返回时间戳和数据来源。
- 日历与日程：查询今天安排、下一事项；新增或修改事项需要明确确认并受权限限制。
- 用药计划：播报 caregiver/provider 已配置的计划，记录 `taken / skipped / unsure / no-response`。
- 提醒：药物、进餐、饮水、活动、预约和自定义提醒；支持重复、稍后提醒、取消和 caregiver 升级规则。
- 常规信息：日期、时间、简单知识问答和陪伴聊天。
- 任务交接：无法安全完成的请求转为 caregiver 待办，而不是编造结果。

安全规则：

- 任何用药回答都要区分“计划中记录的内容”和“通用信息”。
- 用户提出更改剂量、停药、替代药或出现严重不适时，不执行变更，提示联系 caregiver/provider；紧急症状进入安全流程。
- 工具失败时明确说明数据暂不可用，并允许重试；不得用模型记忆伪造实时结果。
- 日常聊天默认不产生单次疾病分类；可在有同意的前提下提取低风险的交互与依从性特征。

### 4.2 Daily Check-in 每日认知状态采集

目标时长：2–5 分钟。目标频率由 care plan 配置，默认每天一次但允许跳过、延期或因疲劳提前结束。

最小采集域：

- 人物、地点和时间定向；
- 近期事件的叙事连续性与顺序；
- 日常功能与获得帮助的程度；
- 一次温和的同会话延迟回忆；
- 语言流畅性、停顿、修正、轮次和响应延迟；
- 音频、视频、身份和参与度质量；
- 疲劳、拒绝、听力/语言不匹配等混杂因素。

会话规则：

- 每次只问一个问题，允许一次简化或澄清。
- 不向患者说“诊断、测试分数、痴呆概率”。
- 用户拒绝、疲劳、痛苦或重复无响应时立即停止或转为关怀对话。
- 同一个协议必须有 `protocolVersion`；prompt 和模型变更不得静默影响纵向基线。
- 采集完成不等于数据可用；必须先通过身份、音频、语言、覆盖度和质量门控。

输出分层：

1. **Session observation**：本次会话的事实性特征和 QC，不给疾病标签。
2. **Baseline state**：`not_started / building / complete / stale`。
3. **Longitudinal state**：`insufficient_data / stable / change_to_review / priority_review`。
4. **Provider review**：复核结论、处理动作和反馈标签。

禁止把单次 `healthy / dementia` 模型输出直接展示给 patient 或 caregiver。

## 5. 关键用户旅程

### 5.1 首次安装和配对

1. 镜面设备生成持久设备身份并完成麦克风、扬声器、摄像头、时间和网络自检。
2. 后端签发短期配对会话，镜面显示 QR 与 6 位备用码。
3. 已登录的 caregiver 扫码或输入码，选择自己有权限的患者。
4. 后端原子绑定 caregiver、patient 和 device，轮换为设备凭据。
5. 镜面拉取最小患者配置、语言、时区、计划和同意状态。
6. 双端显示成功；绑定替换、解绑和所有权变更写入审计日志。

验收标准：

- 配对码 15 分钟失效、一次性使用、支持速率限制和失败锁定。
- caregiver 不能通过修改 `nurseId/patientId` 绑定无权访问的患者。
- QR 扫描和手动输入都可用。
- 解绑后旧设备凭据立即失效；离线设备恢复联网后也必须收到撤销状态。

### 5.2 唤醒并处理日常请求

1. 端侧低功耗监听目标唤醒词。
2. 命中后给出可见/可听反馈并开启会话。
3. Agent 判断是普通对话还是工具请求。
4. 服务器执行 allowlist 工具，返回结构化结果。
5. Agent 用简短语音解释结果；写入最小必要事件日志。
6. 用户说再见或超时后结束，麦克风回到唤醒状态。

验收标准：

- 目标唤醒词可配置并经过真机噪声测试；不得以临时词代替正式验收。
- 工具调用结果包含来源、时间和执行状态。
- 每个有副作用的操作在执行前复述并获得确认。
- 模型不能直接访问数据库或任意 URL。

### 5.3 用药提醒

1. caregiver 创建结构化 medication plan：显示名称、计划时间、时区、服用说明、升级规则和来源。
2. 调度服务生成 reminder occurrence 并推送至镜面。
3. 镜面播报并接受确认、稍后、跳过、不确定。
4. 结果幂等上传；无响应按配置重试。
5. 达到规则时向 caregiver 生成待办或通知。

验收标准：

- 镜面不能创建未获授权的药物或修改剂量。
- 夏令时、跨时区、设备离线和重复消息不得导致重复确认。
- caregiver 可看到计划来源、最近修改人、提醒记录和未确认状态。

### 5.4 每日 check-in 与上传

1. 调度到点提醒，用户也可主动开始。
2. 会话开始前检查同意、设备质量和身份状态。
3. Realtime Agent 执行版本化协议；设备记录 transcript、时序、QC 和被允许的媒体。
4. 本地 durable outbox 写入完成事件，再向服务端同步。
5. 服务端完成 schema 校验、去重、身份/QC 门控和评估。
6. 更新 feature snapshot、baseline/longitudinal window 和 caregiver 状态。
7. 中高等级变化先进入 reviewer 队列，再按政策生成 caregiver-safe 摘要。

验收标准：

- 相同 `sessionId/idempotencyKey` 重传不生成重复会话。
- 断网、杀进程和重启后待上传记录仍在，并自动重试。
- 评估失败不丢失原始会话；状态显示 `pending / failed / completed / excluded`。
- 数据不合格时输出 `insufficient_data`，而不是默认低风险。

### 5.5 caregiver 查看与处理

caregiver 首页至少展示：

- 今日是否完成 check-in；
- 用药/日程提醒的确认与异常；
- 设备最后在线、麦克风/摄像头和上传队列状态；
- 7/30 天依从性与基线完成度；
- provider-approved 的简化状态；
- 待处理任务和通知历史。

所有详情 API 必须根据登录主体做服务端授权，不接受客户端仅凭患者 ID 获取数据。

## 7. 数据与结果要求

每个 session 至少包含：

- `sessionId`, `patientId`, `deviceId`, `programId`；
- `sessionType: companion | screening`；
- `startedAt`, `endedAt`, `timezone`, `language`；
- `consentId`, `consentVersion`, `purpose`；
- `protocolVersion`, `promptVersion`, `modelVersion`, `appVersion`；
- transcript turns 与单调时钟时间戳；
- 音频/视频 artifact reference 或明确的未采集原因；
- QC 分数、flags、identity inclusion；
- processing state 与重试信息；
- feature snapshot reference；
- longitudinal window reference；
- caregiver-safe summary 与 reviewer state。

## 8. 非功能与初始 SLO

以下是工程验收目标，不是临床性能声明：

| 指标 | MVP 目标 |
|---|---|
| 唤醒反馈 | 真机目标唤醒词结束后 P95 ≤ 1.5 秒 |
| Realtime 首音 | 已联网且服务健康时 P95 ≤ 2.5 秒 |
| 天气/日程工具响应 | P95 ≤ 5 秒，超时明确告知 |
| 会话本地落盘 | 每个完成 turn 后 ≤ 1 秒 |
| 上传可靠性 | 可联网后 24 小时内 ≥ 99.5%，幂等无重复 |
| Crash-free sessions | ≥ 99.5% |
| 配对成功率 | 有效账号和网络条件下 ≥ 98% |
| 安全覆盖 | 生产 API 100% 有认证、授权和审计测试 |
| 结果追溯 | 100% 评估具有输入、QC、模型和版本引用 |

临床灵敏度、特异度、校准、误报负担和亚组性能必须由单独的临床验证计划定义，不能以 LLM smoke test 代替。

## 9. 发布门槛

### 家庭技术试用

- P0 安全问题关闭；
- 生产 build 和远程配置冻结；
- 日常工具至少天气和用药提醒端到端可用；
- durable outbox、自动重试和 caregiver 状态闭环；
- 明确同意和隐私撤回；
- 不向 patient/caregiver 展示单次疾病标签。

### 受控研究试点

- protocol、schema、模型和 feature pipeline 版本冻结；
- 身份/QC 门控和 reviewer 工作流可用；
- 数据导出、审计、偏差记录和事件处理可用；
- 完成设备可用性与采集质量研究；
- 预先定义统计分析计划和参考标签。

### 医疗用途

- 只有在预期用途、质量体系、风险管理、临床证据和适用监管路径完成后进入；
- 任何模型或阈值更新都走正式变更控制和再验证。

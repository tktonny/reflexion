## 0. What REFLEXION is

REFLEXION is a **daily reassurance companion** for elderly users (primary target: Singaporean
elderly — Singlish + Mandarin mix). A smart **mirror** device holds a short, natural spoken
check-in each day; a **caregiver** app answers one question — *"is Mum okay today?"* — via a
deterministic, transparent reassurance status. It is explicitly **not** a diagnostic and never
shows a cognitive score.

Three clients, one unified backend, one database:

```
                        ┌──────────────────────────────────────────┐
                        │      reflexion-server  (Node / TS)         │
   Mirror (Android) ───▶│  Express · /api/v1 (canonical)             │
   Expo/RN, device auth  │            /api/*   (legacy, Sunset)       │◀─── Caregiver app (iOS/Android)
                        │  pm2: reflexion-api (:3001) + worker        │      Expo/RN, human JWT
   Admin web (SPA) ─────▶│  JWT/device auth · idempotency · outbox    │
   Vite/React, human JWT │  monitoring pipeline · object store         │◀─── (Qwen realtime: mirror↔Qwen direct WS)
                        └───────────────────┬────────────────────────┘
                                            │
                                   MongoDB Atlas (mongodb+srv)
```

- **Mirror ↔ Qwen:** the realtime conversation streams **directly** from the mirror to Qwen
  (omni realtime WS); the backend only mints a short-lived session ticket and ingests the
  session afterward. The backend is not in the per-utterance audio path.
- **Deployment:** Aliyun ECS + BT-Panel + nginx reverse proxy. API at
  `reflexion.production.tktonny.top` → `127.0.0.1:3001`; admin SPA (static) at
  `admin.reflexion.production.tktonny.top`. See **Part A §3**.

### 2. Key End-to-End Flows

#### (a) Device provisioning → pairing → credential exchange

Sources: `mirror-app/app/index.tsx`, `mirror-app/src/api/devicePairing.ts`, `mirror-app/src/storage/deviceCredentials.ts`, `caregiver-app/app/mirror-management/add.tsx`, `reflexion-server/src/v1/routes/devices.ts`.

1. **Provision (one-time, CLI)** — operator runs `npm run provision:device -- --serial=… --hardware=… --software=…` (`reflexion-server` `scripts/provisionDevice.ts`), producing the **bootstrap credential** (a `bootstrap`-audience JWT holding `did` + `serialHash`). It is planted on the mirror (SecureStore or `EXPO_PUBLIC_DEVICE_BOOTSTRAP_TOKEN`), validated by `getBootstrapCredential()` (`deviceCredentials.ts:32-65`).
2. **Mirror starts a pairing** — boot detects unpaired state and calls `createDevicePairing()` → **`POST /api/v1/device-pairings`** with header `X-Device-Bootstrap` (+ `Idempotency-Key`), body `{ hardwareRevision, softwareVersion, timezone }` (`devicePairing.ts:30-43`; server `devices.ts:24-55`). Server cancels any prior pending pairings for the device and returns `{ pairingId, displayCode (6 digits), state:'pending', expiresAt, pollAfterSeconds }` (10-min TTL, `devices.ts:16,41-51`). The mirror shows the code + a QR encoding `{ type:'reflexion_device_pairing_v2', pairingId, pairingCode }` (`index.tsx:186-205`).
3. **Caregiver claims it** — in `mirror-management/add.tsx` the caregiver types or scans the code; the legacy `POST /api/nurse-patient-config/mirrors/connect` path bridges to **`POST /api/v1/device-pairing-claims`** (human-authed, `requireActor('human')`), body `{ pairingCode, patientId, mirrorName }` (server `devices.ts:106-170`). Server authorizes the patient (`device:assign`), rate-limits failures (5/10 min), retires prior active assignments for the device, inserts a new active **assignment**, flips the pairing to `state:'paired'`, mints an **exchange ticket** (5-min TTL), and emits a `device.paired` outbox event.
4. **Mirror polls status** — every ~3.5 s it calls `getDevicePairing(pairingId)` → **`GET /api/v1/device-pairings/:pairingId`** with `X-Device-Bootstrap` (`index.tsx:99-116`; server `devices.ts:57-104`). When `state==='paired'` it returns `patientDisplayName` + `exchangeTicket` (+expiry). If the device wasn't polling within the initial window, the server **re-issues** the exchange ticket to the owning device on read (bound to `bootstrap.did`) so it can't get stuck "paired but no ticket" (`devices.ts:79-99`). Expired/cancelled → mirror requests a fresh pairing code.
5. **Credential exchange** — `exchangeDeviceCredential()` → **`POST /api/v1/device-credentials/exchange`** with `X-Device-Bootstrap`, body `{ pairingId, exchangeTicket }` (`devicePairing.ts:54-65`; server `devices.ts:172-180,314-341`). Server verifies+consumes the ticket in a transaction, rotates prior device credentials to `rotated`, inserts an active credential, and returns `{ deviceId, credentialId, patientId, accessToken (15 min), accessTokenExpiresAt, refreshCredential (90 d), refreshCredentialExpiresAt }`, persisted by `persistDeviceCredential` (`deviceCredentials.ts:67-81`).
6. **Fetch configuration** — `refreshAndPersistDeviceProfile()` → **`GET /api/v1/devices/:deviceId/configuration`** via `deviceFetch` (Bearer) (`index.tsx:207-219`; server `devices.ts:232-249`). Returns `{ deviceId, configVersion, desired, patient:{ patientId, displayName, preferredLanguage, timezone, carePlan:{ communicationPreferences, dailyRoutine } } }`. Mirror persists profile + language + timezone and routes to `/conversation`. (`GET /api/v1/devices/:deviceId/configuration` is re-polled on every conversation entry to re-validate pairing — `conversation.tsx:550-565`.)
- **Ongoing**: access token auto-rotates via **`POST /api/v1/devices/:deviceId/credential-rotations`** (`deviceCredentials.ts:129-138`); mirror sends **`POST /api/v1/devices/:deviceId/heartbeats`** (`devices.ts:251-282`); caregiver unlink ultimately maps to **`POST /api/v1/devices/:deviceId/revocations`** (`devices.ts:212-230`).

#### (b) Mirror conversation / session lifecycle

Sources: `mirror-app/src/api/sessionSync.ts`, `mirror-app/app/conversation.tsx`, `reflexion-server/src/v1/routes/sessions.ts`.

1. **Begin session** — `beginMirrorSession(type, language, dailyPlan)` first abandons any leftover session, then **`POST /api/v1/sessions`** (Bearer + `Idempotency-Key`), body `{ type: 'companion'|'daily_checkin'|'device_test', clientSessionId, requestedLanguage, clientContext:{ app:'mirror', transport, protocolVersion, reminiscenceEnabled, medicationOccurrenceId } }` → `{ sessionId, stateVersion }` (`sessionSync.ts:152-180`; server `sessions.ts:30`). Session starts in state `created`.
2. **Realtime ticket** — for realtime voice the mirror calls `getActiveQwenTicket()` → **`POST /api/v1/sessions/:sessionId/realtime-tickets`** (`{}` body) → `{ ticket, expiresAt }` (`sessionSync.ts:196-208`; server `sessions.ts:76-93`). Device-only; server mints a short-lived Qwen realtime ticket and flips the session `created→active`. (Cached until ~60 s before expiry.)
3. **Conversation runs on device** — turn-taking / realtime handled by `useConversation` + hooks (`useDirectRealtimeConversation`, `useQwenRealtimeConversation`, `useTurnBasedConversation*`); transcript turns and per-turn timing are accumulated as telemetry (`conversation.tsx:131-148`). End is triggered by goodbye detection, tap, or wake-word off.
4. **Build completion payload** — on end, `finalize()` stops the conversation, captures telemetry + a synthesized WAV artifact, and `buildPendingSessionCompletion()` assembles `transcript_turn` + `capture_metric` events and an `acquisitionSummary` (duration, patientTurns, patientSpeechMs, medianResponseLatencyMs, sessionStartMinuteOfDay, timezone, appVersion, etc.) (`conversation.tsx:365-405`; `sessionSync.ts:214-320`).
5. **Upload event batches** — `uploadPendingSessionCompletion()` posts events in ≤100-item batches to **`POST /api/v1/sessions/:sessionId/event-batches`** (each with its own `Idempotency-Key`) (`sessionSync.ts:322-330`; server `sessions.ts:95-132`, dedupes by eventId/sequence, materializes transcript turns, flips `created→active`).
6. **Upload artifacts (if any)** — `uploadSessionArtifacts()`: **`POST /api/v1/sessions/:sessionId/artifact-upload-plans`** → signed PUT URLs; `PUT` bytes straight to object storage (validated trusted URL); **`POST /api/v1/sessions/:sessionId/artifacts/commit`** (`sessionSync.ts:365-414`; server `sessions.ts:134-201`). Retry path: `POST /api/v1/sessions/:sessionId/upload-retries` (`sessions.ts:245-266`).
7. **Read state + complete** — `GET /api/v1/sessions/:sessionId` to get current `stateVersion`, then **`POST /api/v1/sessions/:sessionId/complete`** with `If-Match: <stateVersion>` and `{ localCompletedAt, finalSequence, artifactIds, acquisitionSummary }` (`sessionSync.ts:332-346`; server `sessions.ts:203-243`). Server optimistic-locks on `If-Match`, verifies event/artifact completeness, flips state to `ingesting`, and emits a `session.completed` outbox event; responds `202 { operationId, state:'queued' }`. Mirror then clears the active session.
8. **Async processing** — the outbox worker runs the monitoring pipeline (§2c / `pipeline.ts:processCompletedSession`). The mirror can poll **`GET /api/v1/sessions/:sessionId/processing-status`** (`sessionSync.ts:360-363`; server `sessions.ts:70`).
9. **Abandon** — a session with no usable transcript (or a client restart) is retired via **`POST /api/v1/sessions/:sessionId/abandon`** `{ reason: 'client_restart'|'user_cancelled'|'transport_failed'|'superseded' }`; idempotent, a `409` is treated as fine (`sessionSync.ts:155-162,350-358`; server `sessions.ts:268-288`).
- **Offline resilience**: completions that can't upload are queued (`src/storage/conversationQueue.ts`) and drained on launch/reconnect by `flushPendingConversations()` (`app/_layout.tsx:22-33`).

#### (c) Reassurance status engine (daily status compute + finalize + caregiver read)

Sources: `reflexion-server/src/v1/monitoring/pipeline.ts`, `reassuranceStatus.ts`, `ruleRegistry.ts`, `reflexion-server/src/v1/routes/monitoring.ts`, `reflexion-server/src/jobs/finalizeDay.ts`.

1. **Session ingestion → features + baselines** — the outbox worker calls `processCompletedSession()` (`pipeline.ts:25`). It runs consent/identity/quality gates (`evaluateConsent`/`evaluateIdentity`/`evaluateSessionQuality`), extracts structured features, optionally embeds transcript text, and — for `daily_checkin` sessions — rebuilds the **operational baseline** (`operationalBaselines`, `baselineType:'reassurance_mvp'`, median+MAD+EWMA over the last 14 sessions, `weeklyRate`) via `rebuildOperationalBaseline` (`pipeline.ts:197-225`). Companion sessions are retained "observation_only" and never move the daily-check-in baseline (`pipeline.ts:110-120`). A separate **longitudinal research baseline** + anomaly score runs in strict shadow isolation and can open a provider review case; it never feeds caregiver colour.
2. **Live status read model** — `computeCaregiverStatus(tenantId, patientId, timezone)` (`monitoring.ts:214-286`) gathers today's completed session, last session, active assignment/device (for `technicalState`), active manual flag, active away period, the away-adjusted `missedStreak`, `weeklyCompletedCount`, and persistence counts from the last 3 finalized `dailyStatuses`. It feeds these into the **pure evaluator** `evaluateReassuranceStatus()` (`reassuranceStatus.ts:94-204`).
3. **Deterministic evaluation** — the evaluator (no DB, no clock, no anomaly/research inputs — structural shadow isolation, `reassuranceStatus.ts:1-6`) produces `status ∈ { establishing, doing_well, worth_checking, needs_attention }` plus a ranked `primaryReason`/`secondaryReasons` and per-metric `metricEvaluations`. Signals: manual flags, missed-day streak M2 (amber≥1/2, red≥3), technical M13 (device issue never framed as decline), and — only when baseline complete + today's session exists — M4 speech / M6 turns / M7 latency / M3 routine window, plus M5 weekly engagement. Single-day amber escalates to red only when the persistence rule is met.
4. **Caregiver read** — caregiver app hits **`GET /api/v1/patients/:patientId/status`** (`monitoring.ts:22-25`, `requireActor('human')` + `authorizePatient('monitoring:read')`), which returns `computeCaregiverStatus(...)` including `status`, `primaryReason`, `secondaryReasons`, `baselineState`, `baselineProgress`, `completedToday`, `technicalState`, `lastInteractionAt`. The app renders it verbatim (never recomputes) via `v1Status.ts` copy tables (`caregiver-app/src/lib/v1Status.ts`) in the dashboard and profile.
5. **Scheduled jobs** — `finalizeDay.ts` runs a once-a-minute idempotent supervisor loop (both gate on local time):
   - `evaluate7pm()` (≥19:00 local) — if no check-in today and not away, queues a deduped notification: `technical_issue` (unreachable), `red_missed_streak` (≥3), or `missed_7pm` (`finalizeDay.ts:48-69`). Dedupe key `patient:localDate:type`.
   - `finalizeDay()` (23:xx or early-morning catch-up) — reuses `computeCaregiverStatus` to upsert the authoritative `dailyStatuses` row (`dailyStatus ∈ completed|missed|technical_issue|away`, `finalStatus` colour, `metricEvaluations`, `missedStreakAfterToday`) so the persisted record and live read model never diverge (`finalizeDay.ts:71-103`).
- Other caregiver monitoring writes: `POST /api/v1/patients/:id/manual-flags`, `POST /api/v1/patients/:id/away-periods` (`monitoring.ts:27-66`); provider-only: `GET /api/v1/patients/:id/monitoring/{summary,timeline,baseline}`, `GET/POST /api/v1/review-cases[...]`.

## Part B — v1 HTTP API (complete reference)


Base path: **`/api/v1`**. All routes below are mounted under this prefix.
Mount order (`src/v1/router.ts:14-22`): identity → patients → devices → sessions → tools → carePlan → monitoring → notifications → admin. All are flat `Router()`s combined by `v1Router.use(...)`, so there are no per-router sub-prefixes — each route's literal path is appended directly to `/api/v1`.

App wiring (`src/app.ts`):
- `requestContext` runs first (assigns/echoes `X-Request-Id`).
- CORS (`src/app.ts:53-61`): allowed request headers `Authorization, Content-Type, Idempotency-Key, If-Match, X-Request-Id, X-Device-Bootstrap`; exposed headers `X-Request-Id, Deprecation, Sunset`; `credentials: true`.

- Body parser `express.json({ limit: '1mb' })`.
- `/api/v1/auth` gets a dedicated auth rate limiter (default 20/min) **layered before** the global `/api/v1` limiter (default 300/min). `auditAccess` middleware sits in front of `v1Router` (`src/app.ts:35-36`).
- 404 for unknown `/api/v1/*` → `ROUTE_NOT_FOUND` (`v1NotFound`, `http.ts:22`), then `apiErrorHandler`.
- Health (NOT under `/api/v1`, no auth): `GET /health`, `GET /healthcheck` → `{ ok: true }`.

There is **no explicit API-version request/response header** other than the `/api/v1` path segment. Legacy API (`ENABLE_LEGACY_API=true`) is mounted after v1 and, when active, adds `Deprecation: true` + `Sunset` headers to legacy responses only.

### 1. Response envelope, errors, pagination

Defined in `src/v1/platform/http.ts`.

#### Success envelope
Every successful JSON response is wrapped:
```json
{ "data": <payload>, "meta": { "requestId": "req_..." } }
```
- `sendData(res, data, status=200)` → `{ data, meta: { requestId } }` (`http.ts:14-16`).
- Some endpoints return **204 No Content** with an empty body (logout) or **201/202** status codes.
- `X-Request-Id` is set on the response for every request; client may supply it (must match `/^[A-Za-z0-9._:-]{8,128}$/`, else server generates `req_<uuid>`).

#### Paginated envelope
`sendPage(res, data[], nextCursor)` → (`http.ts:18-20`):
```json
{ "data": [ ... ], "meta": { "requestId": "req_...", "nextCursor": "<id>|null" } }
```
- Cursor pagination is opaque forward-only on `_id`. Query params: `limit` (1–100, default 25) and `cursor` (`validation.ts:55-60`).
- `nextCursor` is the last item's id when more rows exist, else `null`. List endpoints fetch `limit+1` rows to detect `hasMore`.

#### Error envelope
Thrown `ApiError`s (and unhandled errors coerced to `INTERNAL_ERROR`) are shaped by `apiErrorHandler` (`http.ts:30-48`):
```json
{
  "error": { "code": "STRING_CODE", "message": "…", "retryable": false, "details": [ { "field": "…" } ] },
  "meta": { "requestId": "req_..." }
}
```
- `ApiError(status, code, message, retryable=false, details=[])` (`errors.ts`).
- Helper factories (`errors.ts`): `badRequest`→400 `VALIDATION_FAILED`/custom, `unauthorized`→401 `UNAUTHORIZED`, `forbidden`→403 `FORBIDDEN`, `notFound`→404 `NOT_FOUND`, `conflict`→409 custom.
- Common cross-cutting codes: `UNAUTHORIZED` (401), `INVALID_TOKEN` (401, expired/bad JWT), `FORBIDDEN` (403), `NOT_FOUND`/`ROUTE_NOT_FOUND` (404), `VALIDATION_FAILED`/`INVALID_BODY` (400), `RATE_LIMITED` (429, retryable), `INTERNAL_ERROR` (500, retryable). `retryable` is a boolean hint clients can use for backoff.
- `details` is an array of `{ field, ... }` objects for validation failures (`validation.ts`).

#### Rate limiting (`src/v1/platform/rateLimit.ts`)
- In-memory per-IP fixed window. Headers set on every response: `RateLimit-Limit`, `RateLimit-Remaining`, `RateLimit-Reset` (unix seconds). On exceed → 429 `RATE_LIMITED` (retryable) + `Retry-After` header.
- Two namespaces: `auth` (default 20/min via `AUTH_RATE_LIMIT_PER_MINUTE`) for `/api/v1/auth/*`; `api` (default 300/min via `API_RATE_LIMIT_PER_MINUTE`) for all of `/api/v1`.
- Additional business rate limit: failed pairing claims — 5 failures / 10 min per (tenant,user) → 429 `PAIRING_ATTEMPTS_EXCEEDED` (`devices.ts:404-409`).

#### Idempotency (`src/v1/platform/idempotency.ts`)
- All state-changing POST/PUT that call `executeIdempotent(...)` **require** an `Idempotency-Key` header of 16–128 chars, else 400 `IDEMPOTENCY_KEY_REQUIRED`.
- Keyed by `(actorKey, routeKey, key)` where `actorKey` is `"<kind>:<subjectId>"` (or an explicit key, e.g. `bootstrap:<deviceId>`, `device-refresh:<credentialId>`). Records stored in `idempotency_records`, TTL 24h.
- Replay with the **same body hash** returns the stored status+response. Replay with a **different body** → 409 `IDEMPOTENCY_KEY_REUSED`. A still-processing duplicate → 409 `REQUEST_IN_PROGRESS` (retryable). Some responses containing secrets are sealed via a codec before storage (realtime tickets, upload plans).

#### Optimistic concurrency (`If-Match`)
- Endpoints that mutate versioned resources require `If-Match: <integer version>` (weak/quoted forms tolerated). Missing/invalid → 400 `IF_MATCH_REQUIRED`. Mismatch → 409 `VERSION_CONFLICT`. Applies to: `PATCH /patients/:patientId`, `POST /sessions/:sessionId/complete`, `PUT /patients/:patientId/care-plan` (when a plan already exists), `PATCH /medication-plans/:planId`.

### 2. Authentication & authorization model

Sources: `src/v1/platform/auth.ts`, `tokens.ts`, `principal.ts`, `provisionDevice.ts`.

#### Token kinds & TTLs (`tokens.ts`)
JWT-like HS256 tokens signed with `JWT_SECRET`, `iss:"reflexion"`, `aud:"reflexion-api"`. `verifyAccessToken` checks signature (timing-safe), iss/aud/sub/exp, and expected `kind`. Invalid/expired → 401 `INVALID_TOKEN`.

| Kind | How obtained | Claims | TTL |
|---|---|---|---|
| `human` | `POST /auth/sessions` (access) | `uid, tid, sid, roles[], scopes[]` | access **15 min**; refresh **30 days** (`identity.ts:15-16`) |
| `device` | credential exchange / rotation (access) | `did, pid, tid, cid, roles:['device'], scopes[]` | access **15 min**; refresh credential **90 days** (`devices.ts:18-19`) |
| `bootstrap` | issued offline by `provisionDevice.ts` | `did, serialHash, roles:['device_bootstrap'], scopes:['device:pair']` | **30 days** |

Device access-token scopes (`DEVICE_SCOPES`, `devices.ts:20`): `session:write, session:read, care_plan:read, reminder:respond, device:heartbeat`.

#### Actor guards
- `requireActor(...kinds)` (`auth.ts:8`): validates `Authorization: Bearer <token>`, verifies kind ∈ kinds (default `['human','device']`), and **re-checks live state in Mongo** — human: an `active` `auth_sessions` row with unexpired refresh; device: an `active` `device_credentials` row (unexpired) **and** an `active` `device_assignments` row. On failure → 401 `UNAUTHORIZED`/`INVALID_TOKEN`. Sets `request.principal`.
- `requireHumanRole(...roles)` (`auth.ts:59`): principal must be human and hold at least one of the roles, else 403 `FORBIDDEN`. Used as `requireAdmin = requireHumanRole('operator','tenant_admin')` (admin routes) and `requireHumanRole('provider','reviewer','tenant_admin')` (review routes).
- `getPrincipal(request)` (`auth.ts:54`): returns the `HumanPrincipal | DevicePrincipal`, else throws 401.
- `authorizePatient(request, patientId, scope)` (`auth.ts:71`): loads the tenant-scoped, non-archived patient. Device → must match its own `patientId` and hold `scope` in its token scopes. Human → `tenant_admin` bypasses; otherwise an `active` `care_relationships` row for (tenant,patient,user) whose `scopes` array contains `scope` and is time-valid. Failure → 403 `FORBIDDEN`.
- **Bootstrap auth** is NOT via `requireActor`; device-pairing/exchange routes call `bootstrapClaims(request)` (`devices.ts:284`) which reads the **`X-Device-Bootstrap`** header, verifies a `bootstrap` token, and confirms the device exists and is not revoked.

#### Relationship scopes (care_relationships)
Seen scope strings used with `authorizePatient`: `patient:read`, `patient:write`, `device:assign`, `care_plan:read`, `care_plan:write`, `monitoring:read`, `session:read`, `session:write`, `reminder:respond`, `review:read`, `review:write`. Default scopes granted to the creating caregiver (`patients.ts:13-15`): `patient:read, patient:write, device:assign, care_plan:read, care_plan:write, monitoring:read`.

#### Human roles seen
`caregiver` (default), `operator`, `tenant_admin`, `provider`, `reviewer`. (`operator|tenant_admin` = admin; `provider|reviewer|tenant_admin` = clinical review.)

**Auth column legend used below:** `public` (no auth) · `bootstrap` (X-Device-Bootstrap) · `device` (device Bearer) · `human` (human Bearer) · `human+device` (either) · role suffix (e.g. `human+admin`, `human+reviewer`).

## Part C — Legacy API, Data Model & Platform / Security


Scope: `reflexion-server/`. All file:line citations are relative to `reflexion-server/src/`.
Two API surfaces coexist in one Express app (`app.ts`): the **v1 platform** (`/api/v1/*`, always on) and the
**legacy caregiver API** (root-mounted, gated by `ENABLE_LEGACY_API`). The legacy surface has been re-plumbed to
read/write v1 collections where possible (see `lib/legacyV1Bridge.ts`, `lib/v1Conversations.ts`); the notes call out
which endpoints still touch the old `NursePatientConfig`-family collections.

### (1) LEGACY API

#### Mounting & lifecycle gates (`app.ts`)

- v1 is always mounted: `app.use('/api/v1', rateLimit(...), auditAccess, v1Router)` (`app.ts:36`). Auth sub-limit on

### (3) PLATFORM & SECURITY MODEL

#### Token types & TTLs

JWTs are HS256, HMAC-signed with `JWT_SECRET` (`tokens.ts:31-47`); fixed `iss:'reflexion'`, `aud:'reflexion-api'`; claims include
`sub`, `kind`, and role-specific ids (`tid/uid/did/pid/sid/cid`, `roles`, `scopes`, `serialHash`) (`tokens.ts:7-23`).
`verifyAccessToken` does timing-safe signature comparison, checks `iss/aud/sub/exp` and optional `expectedKinds` (`tokens.ts:49-68`).

| Token | Kind | TTL | Constant / source | Transport |
|---|---|---|---|---|
| Device bootstrap | `bootstrap` | provisioning-time (long-lived; carries `did`+`serialHash`) | issued by `scripts/provisionDevice.ts`; verified `devices.ts:284-292` | `X-Device-Bootstrap` header |
| Device access | `device` | **15 min** | `ACCESS_TTL_SECONDS = 15*60` (`devices.ts:18`) | `Authorization: Bearer` |
| Device refresh credential | (secret, not JWT) | **90 days** | `REFRESH_TTL_MS = 90*24*60*60*1000` (`devices.ts:19`) | request body |
| Human access | `human` | **15 min** | `ACCESS_TTL_SECONDS = 15*60` (`identity.ts:15`) | `Authorization: Bearer` |
| Human refresh token | (secret, not JWT) | **30 days** | `REFRESH_TTL_MS = 30*24*60*60*1000` (`identity.ts:16`) | request body |
| Pairing code | (6-digit) | **10 min** | `PAIRING_TTL_MS = 10*60*1000` (`devices.ts:16`) | display code |
| Exchange ticket | (secret) | **5 min** | `EXCHANGE_TTL_MS = 5*60*1000` (`devices.ts:17`; also `legacyV1Bridge.ts:14`) | pairing poll / claim |
| Password reset token | (secret) | **30 min** | `identity.ts:72` / legacy `TOKEN_TTL_MS` (`auth/password-reset.ts:13`) | email link |
| Qwen realtime ticket | (upstream DashScope) | 60s..3600s (default 900/1800s) | `qwen.ts:9`, `qwen-token.ts:16` | session ticket |

Device scopes granted on credential issue: `['session:write','session:read','care_plan:read','reminder:respond','device:heartbeat']`
(`DEVICE_SCOPES`, `devices.ts:20`).

#### Principals & authorization (`auth.ts`, `principal.ts`)

- Two principal kinds (`principal.ts`): **`HumanPrincipal`** {userId, tenantId, sessionId, roles[], scopes[]} and
  **`DevicePrincipal`** {deviceId, credentialId, tenantId, patientId, roles:['device'], scopes[]}.
- `requireActor(...kinds)` (`auth.ts:8-52`): parses Bearer JWT, then **re-validates against the DB** — human tokens require an
  active `auth_sessions` row with unexpired refresh; device tokens require both an active `device_credentials` row (unexpired
  refresh) **and** an active `device_assignments` row. This means revoking assignment/credential immediately invalidates a still-valid
  access token (the mechanism the legacy unlink relies on).
- `requireHumanRole(...roles)` (`auth.ts:59-69`): enforces a role on human principals.
- `authorizePatient(request, patientId, scope)` (`auth.ts:71-95`): loads the patient scoped to the principal's tenant; a device
  principal must own that patient and hold the scope; `tenant_admin` humans pass; otherwise requires an active
  `care_relationships` edge carrying the scope and unexpired `validTo`.

#### Cryptography (`crypto.ts`)

- `requireServerSecret(name)` — enforces env secret ≥32 chars (`crypto.ts:26-32`).
- `hmac(value, PAIRING_PEPPER)` — HMAC-SHA256 used for **pairing code hashing** (`codeHash`, `crypto.ts:7-9`; keyed by
  `PAIRING_PEPPER`). Pairing lookups match `hmac(code)` so codes are never stored in the clear.
- `sealSecret`/`openSecret` — AES-256-GCM, key = `sha256(CREDENTIAL_ENCRYPTION_KEY)`, random 12-byte IV, GCM tag; serialized
  `iv.tag.ciphertext` (base64url) (`crypto.ts:34-53`). Used to store the exchange ticket cipher and to seal password-reset /
  realtime-ticket payloads across the outbox.
- `hashSecret`/`verifySecret` — salted SHA-256 (`sha256$<salt>$<digest>`), timing-safe compare (`crypto.ts:11-24`). Used for
  refresh credentials, exchange ticket hash, reset token hash.
- `sha256(value)` — hex digest, used as a fast **lookup digest** (`secretDigest`, `tokenDigest`, `refreshDigest`, exchange
  `Digest`, idempotency `requestHash`) alongside the salted verify hash.
- Human/legacy **passwords** use a separate pbkdf2 scheme (`lib/password.ts`: `pbkdf2_sha256$120000$salt$hash`, 120k iterations).

#### Idempotency (`idempotency.ts`)

`executeIdempotent(request, routeKey, execute, explicitActorKey?, codec?)`:
- Requires `Idempotency-Key` header, 16–128 chars (`idempotency.ts:20-23`).
- `actorKey` = `explicitActorKey` or `<kind>:<subjectId>`; unique key = `{actorKey, routeKey, key}` inserted into
  `idempotency_records` with a 24h `expiresAt` (`idempotency.ts:33-35`). Duplicate-key race (Mongo 11000) falls back to replay.
- `requestHash = sha256(body)`. Replaying with a **different** body → 409 `IDEMPOTENCY_KEY_REUSED`; replay while still
  `processing` → 409 `REQUEST_IN_PROGRESS`; completed replay returns stored `{status,data}` (`idempotency.ts:58-66`).
- Failed executions delete the processing record so the client can retry (`idempotency.ts:52-55`). Optional `codec` lets
  sealed payloads (e.g. realtime tickets, `sessions.ts:90`) be encrypted at rest in the ledger.
- Bootstrap-authenticated device routes pass an explicit actor key (e.g. `bootstrap:${did}`, `devices.ts:52`).

#### Rate limiting (`rateLimit.ts` + route-level)

- In-memory sliding bucket keyed by `<namespace>:<ip>`, 60s window, emits `RateLimit-Limit/Remaining/Reset` and 429 +
  `Retry-After` (`rateLimit.ts`). Buckets GC'd when the map exceeds 10k entries.
- Wired in `app.ts:35-36`: `auth` namespace = `AUTH_RATE_LIMIT_PER_MINUTE` (default **20/min**) on `/api/v1/auth`; `api`
  namespace = `API_RATE_LIMIT_PER_MINUTE` (default **300/min**) on `/api/v1`. In-process only (not shared across instances).
- Domain rate limit: pairing-claim failures are counted from `audit_events` — ≥5 `pairing.claim_failed` in a `PAIRING_TTL_MS`
  window → 429 `PAIRING_ATTEMPTS_EXCEEDED` (`devices.ts:404-418`); pairings also carry a `failedAttempts < 5` guard.

#### Outbox pattern (`outbox.ts` + `workers/outboxWorker.ts` + `jobs/outboxWorker.ts`)

- `appendOutbox(db, input, session?)` inserts an `outbox_events` doc within the same transaction as the state change
  (`_id: evt_…`, `state:'pending'`, `attempt:0`, `nextAttemptAt:now`) (`outbox.ts:16-34`). Producers: device paired/revoked/heartbeat
  (`devices.ts`), session created/completed, features.extracted, embedding.created, baseline.rebuilt, anomaly.scored,
  review_case.created (`pipeline.ts`), password_reset.requested (`identity.ts:77`).
- Worker `processNextOutboxEvent` (`workers/outboxWorker.ts`): atomically leases the oldest due event
  (`findOneAndUpdate`, `state→processing`, `leaseUntil = now+60s`, `attempt++`, sort `occurredAt` asc). Per-consumer dedupe via
  `event_consumptions` (consumer `platform-v1-worker`). On failure: exponential backoff `min(2^attempt*1000, 15min)`, and after
  `MAX_ATTEMPTS = 8` → `dead_letter`; a failed `session.completed` also marks the session `processing_failed`
  (`workers/outboxWorker.ts:35-56`).
- `handleEvent` dispatch (`workers/outboxWorker.ts:78-98`): `artifact.committed`→`verifyArtifact`,
  `session.completed`→`processCompletedSession`, `medication_plan.changed`→`materializeMedicationReminders`,
  `password_reset.requested`→`sendPasswordResetEmail` (opens sealed token), `review_case.created`→`materializeReviewCaseNotifications`.
- Runner `jobs/outboxWorker.ts`: loop polling every 1s (or `--once`), clean shutdown on SIGINT/SIGTERM.

#### Object store drivers (`objectStore.ts`, `objectStoreLocal.ts`)

- `getObjectStore()` selects by `OBJECT_STORE_DRIVER`: `s3` → `S3CompatibleObjectStore`; `local` → `LocalObjectStore`; else
  `UnconfiguredObjectStore` (all ops 503 `OBJECT_STORE_NOT_CONFIGURED`) (`objectStore.ts:23-37`).
- Common contract (`ObjectStore`): `prepareUpload()` returns a short-lived signed **PUT** URL + `requiredHeaders` + `expiresAt`
  (clamped 60s..3600s); `verify()` confirms stored size + SHA-256. Consumed by session artifact commit
  (`sessions.ts:251-259`) and `verifyArtifact` (`pipeline.ts:13-22`).
- **S3 driver**: manual AWS SigV4 presign (`objectStore.ts:71-95`); requires `OBJECT_STORE_ENDPOINT/BUCKET/ACCESS_KEY/SECRET_KEY`
  (+ region, optional SSE `AES256`/`aws:kms` + KMS key). `verify()` HEADs the object and matches `content-length` +
  `x-amz-meta-sha256`. HTTPS enforced in production (`objectStore.ts:119-125`).
- **Local driver**: signed PUT URL points back to **this** server at `/api/v1/object-store/objects` with query
  `key/exp/sig/sha256` (HMAC via `OBJECT_STORE_LOCAL_SECRET`, ≥16 chars). The PUT target is mounted **before** `express.json`
  so the raw body is not parsed, and outside the auth stack since the signed URL carries its own token (`app.ts:24`,
  `objectStoreLocal.ts:70-97`). Guards: expiry, timing-safe signature, SHA-256 header match, path-traversal rejection, max
  64 MB (`OBJECT_STORE_LOCAL_MAX_BYTES`). Not for multi-instance production (local disk).

#### Monitoring pipeline & rule versions

- Pipeline (`monitoring/pipeline.ts`) processes each `session.completed` event through stages:
  consent gate → identity (device-assignment only, **not biometric**, `pipeline.ts:155-168`) → quality gate → feature
  extraction → optional embedding → monitoring/baseline/anomaly. Companion & non-`daily_checkin` sessions are stored
  `observation_only` and never mutate the validated baseline or open review cases (`pipeline.ts:112-120`).
- Versions:
  - `PIPELINE_VERSION` = `MONITORING_PIPELINE_VERSION` env or **`'longitudinal-v1'`** (`pipeline.ts:10`) — stamped on
    feature snapshots, identity links, quality assessments, processing runs, baseline models.
  - `RULE_VERSION` = `MONITORING_RULE_VERSION` env or **`'transparent-rules-v1'`** (`pipeline.ts:11`) — stamped on anomaly
    scores (`scoreVersion`), monitoring windows (`ruleVersion`), research baseline `algorithmVersion`.
  - Operational baseline `algorithmVersion` = **`'reassurance-median-mad-ewma-v1'`**, `alpha 0.1` (`pipeline.ts:214`).
  - Reassurance status engine rules: `ruleVersion` = **`'reassurance-rules-v1'`** (`monitoring/ruleRegistry.ts:33`), a frozen
    threshold set (baseline windowDays 14 / minSessions 7 / ewmaAlpha 0.1; M4/M6 speech+turns; M7 latency; M3 routine window;
    M5 weekly engagement; M1/M2 missed streak amber@2/red@3; technical unreachable@15min). Overridable at runtime by a
    `rule_registry` overlay shallow-merged over defaults (`ruleRegistry.ts:50-62`), consumed by
    `monitoring/reassuranceStatus.evaluateReassuranceStatus` (`reassuranceStatus.ts:94-203`).
- Two baseline families: **operational** (`reassurance_mvp`, median/MAD/EWMA, `pipeline.ts:197-225`) and **longitudinal
  research** (`longitudinal_research`, scalar aggregates + embedding centroid/cosine-distance, `pipeline.ts:239-314`). Anomaly
  band with persistence count → `review_recommended`/`priority_review` opens a `review_cases` row (`pipeline.ts:284-368`).
- Scheduled jobs (`jobs/finalizeDay.ts`) reuse `computeCaregiverStatus` (`routes/monitoring.ts:214`) so the persisted
  `daily_statuses` row and the live read model cannot diverge: `evaluate7pm` queues deduped 19:00 missed/technical/red-streak
  notifications; `finalizeDay` writes the authoritative `daily_statuses` at ~23:59 local. Notifications dedupe on
  `<patientId>:<localDate>:<type>` (`finalizeDay.ts:37-46`).
- Realtime ticket minting (`platform/qwen.ts`): `createQwenRealtimeTicket` mints an upstream DashScope token
  (`QWEN_API_KEY`/`DASHSCOPE_API_KEY`, `QWEN_BASE`), returns endpoint (`QWEN_REALTIME_ENDPOINT`, default
  `wss://dashscope.aliyuncs.com/api-ws/v1/realtime`) + sessionPolicy (model default `qwen3.5-omni-flash-realtime`,
  `clientMaySelectModel:false`, `clinicalDiagnosisAllowed:false`). Only an assigned device may request it (`sessions.ts:76-93`),
  and the response is sealed in the idempotency ledger via a codec.

### Known gaps / follow-ups
- **Push notifications (FCM) are not implemented end-to-end.** The backend has no push-send or device-token-registration path, and the caregiver app ships no `google-services.json` (`android.googleServicesFile` unset). Android background push **requires Google FCM** (a platform requirement, independent of Expo). The app currently uses an Expo push token (`getExpoPushTokenAsync`), which relays via Expo's servers — to keep everything on own infrastructure, switch to a native FCM device token + backend → FCM HTTP v1 directly. **Verify FCM reachability from the Aliyun (China-region) server first.**
- **The caregiver "Support" chatbot is a Wizard-of-Oz** — it POSTs to legacy `/api/feedback` and is **not** wired to the admin support-thread surface (`/api/v1/support/threads` ↔ `/api/v1/admin/support/threads`). Connecting the two is a follow-up.
- **The mirror detects unlink/revocation only at conversation-screen mount** (`verifyStoredPairing` → `/configuration` 403 → back to pairing). There is no live re-check while a conversation is running, so a remote unlink takes effect on the next app restart. Optional enhancement: periodic re-verification (piggyback the device heartbeat).
- **The backend logs only non-`ApiError` exceptions** (`src/v1/platform/http.ts`); 4xx business errors (e.g. `EXCHANGE_TICKET_INVALID`) are returned but not logged, so operational debugging often needs a direct DB query. Consider structured request/error audit logging.
- **The backend `.env` is not committed** and lives only on the server (loaded via dotenv). It was lost once during the server rename and restored from a local copy — keep a secure backup or move to a secrets manager.
- **Legacy `/api/*` is gated by `ENABLE_LEGACY_API=true`** (Sunset). The caregiver app still relies on legacy sign-in + `/api/nurse-patient-config/*`; migrate it fully onto v1 before disabling legacy.
- **Five declared-but-unused collections** (`reminder_rules`, `session_observations`, `notification_suppressions`, `feature_registry`, `model_registry`) are reserved and not yet written.
